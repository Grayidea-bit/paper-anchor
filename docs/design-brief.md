# Paper Anchor — UI/UX 重新設計簡報（Design Brief）

> 給設計者（人類或 AI）的完整脈絡。目標：在不破壞既有互動契約的前提下，重新設計整體視覺與體驗。

## 1. 產品是什麼

**Paper Anchor**：與 LLM「在同一篇文獻上」共讀的雙欄 PDF 閱讀器。開源、自架、單人使用。

**一句話**：Every answer, anchored to the source——AI 回答中的每個論斷都是可點擊的引用，一鍵跳回 PDF 原文並精準高亮。

**核心差異（設計必須放大的東西）**：
1. **引用錨點**是產品靈魂。回答裡的引用 chip → 點擊 → 左欄 PDF 跳頁 + 琥珀色高亮框住原文區塊。「可驗證」的感覺要貫穿整個設計語言。
2. **選取提問**：在 PDF 圈選文字 → 浮動選單（解釋/翻譯/質疑/提問）。
3. **三層問答**：單篇文獻、專案（一組文獻）、全庫。跨文獻引用點擊會自動切換左欄的 PDF。

## 2. 使用者與情境

研究生／跨領域工程師，一次讀 1–3 小時，每週 3–10 篇論文。桌面環境（最低寬 1280px），常開著超過一小時——**低疲勞、可長時間停留**比酷炫重要。雙語使用者（介面與回答語言可一鍵切 zh-TW/en，設計需容納兩種文字長度）。

## 3. 資訊架構與畫面清單

```
App
├─ Header（常駐）：品牌字標｜← 返回文獻庫｜當前文獻標題｜中/EN 切換｜連線狀態點
├─ 左欄（58%）
│   ├─ A. 文獻庫視圖（未開啟文獻時）
│   └─ B. PDF 閱讀器（開啟文獻時）
└─ 右欄（42%）
    ├─ C. 對話面板（三種範圍）
    └─ D. 導讀卡（僅單篇對話，置頂）
```

### A. 文獻庫（左欄）
- 頂部動作：上傳 PDF、全庫問答
- **專案分組**：每組＝標題＋文獻數＋「專案問答」按鈕＋改名/刪除；組內文獻卡片
- 文獻卡片：襯線標題、頁數、狀態（已上傳→解析中→建立索引→產生導讀→可閱讀｜失敗+原因）、專案指派下拉、刪除
- 「未分類」固定最後一組；＋新專案（inline input）
- 底部：累計 token 用量（in/out）
- 狀態：空庫、上傳中、處理中（輪詢刷新）、失敗

### B. PDF 閱讀器（左欄）
- 逐頁 canvas 渲染 + 透明文字層（可選取）；頁面虛擬化（未進視野的頁是佔位框）
- **高亮層**：琥珀色半透明框 + 邊框 + 出現時脈衝動畫（引用跳轉的落點）
- 頁碼標示；選取文字後的**浮動選單**（解釋/翻譯/質疑/提問，錨定在選取上方）
- 狀態：載入中、載入失敗

### C. 對話面板（右欄）
- **範圍列**（專案/全庫對話才有）：顯示「專案：{名}」或「範圍：全部文獻」＋關閉鈕；單篇對話無此列
- 訊息流：目前是期刊訪談式 Q/A 排版（氧化紅 Q／墨色 A、條目間細線）——可重新想像，但「非泡泡、易長文閱讀」的方向經過驗證
- **引用 chip**：內文中的小標籤。同文獻＝outline 樣式；**跨文獻＝填色樣式**，hover 顯示來源文獻標題+頁碼。這是全產品最重要的微元件
- 使用者訊息可帶「選取引文」（琥珀左框斜體引文塊）
- assistant 訊息是 markdown（粗體/列表/表格/程式碼）
- 串流狀態：思考中（三點動畫，推理模型可能 20–40 秒才吐字——這段等待的體驗值得設計）、逐字輸出
- 錯誤列＋重試按鈕；輸入區：＋新對話、textarea（Enter 送出）、送出鈕
- 附掛選取列（自由提問時顯示在輸入區上方，可移除）

### D. 導讀卡（單篇對話置頂）
- 折疊式；tldr 斜體引言 + 五節（研究問題/方法/主要發現/貢獻/限制），每節附 p.X 引用 chip 可跳轉
- 狀態：產生中（+重新產生按鈕）、完成

## 4. 關鍵互動流程（不可破壞的契約）

1. **引用跳轉**：點 chip → 左欄捲動到目標頁 → 高亮動畫。跨文獻時先換 PDF 再跳。使用者需要清楚感知「我被帶到哪了」。
2. **選取提問**：圈選 → 浮動選單 → 預設動作直接送出（帶引文）／自由提問附掛到輸入區。
3. **範圍切換**：文獻庫 → 專案問答/全庫問答 → 對話範圍改變但左欄不變；點跨文獻引用 → 左欄換 PDF、右欄對話不動。
4. **語言切換**：Header 的中/EN 同時切介面文字與 AI 回答語言。

## 5. 現有設計（可保留、可推翻）

目前是「書齋/學術期刊」風：紙色底（#f6f1e7 系）、墨色文字、氧化紅 #8c2f24 唯一強調色、琥珀 #e8b931 高亮；Source Serif 4 + Noto Serif TC 標題、IBM Plex Mono 元資料、Noto Sans TC 內文。

- **必須保留的語意**：高亮=琥珀系（與 PDF 選取色一致）；引用 chip 的「同文獻 vs 跨文獻」視覺區分；Q/A 的視覺層級分明。
- **可自由重新想像**：整體色彩方向（含深色模式——長時間閱讀的真實需求，目前沒有）、字型組合、版面密度、動效語言、文獻庫的組織視覺（分組 vs 側欄 vs 其他）。

## 6. 技術約束

- React + TypeScript + CSS Modules；**無 UI 元件庫**，樣式全手寫；主題色一律走 CSS 變數（`:root` design tokens），這是換膚的介面。
- PDF 頁面是 canvas（不可 CSS 重繪內容），高亮/文字層是絕對定位 overlay——PDF 區的設計自由度在「周圍」（背景、陰影、頁碼、捲軸），不在頁面本身。
- Google Fonts 可用；桌面優先，最低 1280px，無需行動版。
- 雙語：所有文案經 i18n，中英文長度差異大（如「專案問答」vs「Ask project」），元件需彈性。
- 串流渲染：訊息內容逐 token 更新，避免依賴 layout 會跳動的設計。

## 7. 期望交付

1. **Design tokens**：色彩（含深色模式）、字型階層、間距、圓角、陰影——以 CSS 變數格式。
2. **元件規格**：引用 chip（4 態：同文獻/跨文獻/解析中/hover）、訊息條目、文獻卡片、狀態標籤、浮動選單、範圍列、導讀卡。
3. **三個關鍵畫面的完整版面**：文獻庫、單篇閱讀+對話、專案問答+跨文獻引用。
4. 等待體驗提案：推理模型 20–40 秒思考期的設計。
5. （加分）深色模式全套。
