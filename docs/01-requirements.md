# 需求分析 — AI 文獻導讀（Paper Reader）

> 角色：需求分析師 / PM
> 狀態：v1.0（2026-07-04）

## 1. 產品定位

**一句話**：讓使用者與 LLM「在同一篇文獻上」互動——LLM 看得到你正在讀的段落，你看得到 LLM 引用的原文位置。

**核心問題**：現有做法（把 PDF 丟進 ChatGPT / 複製貼上段落）的痛點：
- P1 對話與原文脫節：LLM 說「第三節提到…」，使用者要自己翻回去找。
- P2 提問成本高：想問某一段，得先複製貼上，還要補上下文。
- P3 回答不可驗證：LLM 的摘要無法對照原文，幻覺難以察覺。
- P4 長文獻超出上下文：整篇貼入品質差、費用高。

**解法核心**：雙欄介面（左：PDF 閱讀器｜右：對話），以「引用錨點」雙向連動。

## 2. 目標使用者（Persona）

| Persona | 情境 | 關鍵需求 |
|---|---|---|
| 研究生（主要） | 每週讀 3–10 篇論文，需快速判斷是否精讀 | 結構化導讀、名詞解釋、引用可驗證 |
| 跨領域工程師 | 讀不熟領域的論文找可落地方法 | 白話翻譯、方法段落深挖 |
| 大學生 | 課堂指定閱讀，英文文獻吃力 | 翻譯、逐段解說 |

## 3. 使用者故事與功能需求

### MVP（P0，必須）
- **F1 文獻上傳**：上傳 PDF（≤50MB），解析文字與版面座標，失敗要有明確錯誤訊息。
- **F2 雙欄閱讀介面**：左側 PDF 原樣渲染（PDF.js），右側對話面板；狀態各自獨立捲動。
- **F3 自動導讀**：上傳完成後自動產生結構化導讀卡：研究問題／方法／主要發現／貢獻／限制，每一項附引用錨點。
- **F4 文獻內對話（RAG）**：針對該篇文獻提問；檢索相關 chunk 作為上下文；回答串流輸出。
- **F5 引用錨點（雙向連動）**：
  - LLM 回答中的引用（如 `[p.3 §2.1]`）可點擊 → 左側跳頁並高亮原文。
  - 這是本產品與「把 PDF 丟給 ChatGPT」的差異核心，不可砍。
- **F6 選取提問**：左側選取文字 → 浮出快捷選單（解釋 / 翻譯 / 質疑 / 自由提問）→ 選取內容連同前後文自動帶入對話。
- **F7 對話歷史**：每篇文獻各自保存多個對話串，重新開啟可續聊。

### P1（MVP 後第一批）
- F8 文獻庫：列表、搜尋、刪除、閱讀進度。
- F9 名詞卡片：hover 專有名詞顯示解釋（以文獻上下文為準）。
- F10 匯出：導讀 + 對話匯出 Markdown。
- F11 多語系：介面與回答語言可設定（預設繁中回答、保留原文術語）。

### P2（暫不做，記錄避免架構堵死）
- 多篇文獻交叉比較、arXiv/DOI 直接匯入、協作共讀、瀏覽器外掛。

## 4. 非功能需求

| 項目 | 要求 |
|---|---|
| 效能 | 上傳→可對話 ≤ 30s（20 頁內）；首 token ≤ 3s（串流） |
| 成本 | 單篇導讀 + 20 輪對話的 LLM 成本可估算、可記錄（token 用量入庫） |
| 可靠性 | PDF 解析失敗不可讓系統崩潰；掃描版 PDF 明確告知不支援（MVP 不做 OCR） |
| 安全 | 使用者僅能存取自己的文獻；檔案不外流至 LLM 供應商以外 |
| 可攜性 | docker compose up 一鍵啟動；LLM 供應商可抽換（預設 DeepSeek，介面相容 OpenAI 格式） |

## 5. 明確不做（Out of Scope for MVP）

- OCR 掃描版 PDF、非 PDF 格式（Word/HTML/EPUB）
- 帳號系統做完整（MVP 用單一使用者或簡易 session 即可，但資料模型預留 user_id）
- 行動裝置版面（桌面優先，最低寬度 1280px）

## 6. 成功指標（MVP 驗收）

1. 上傳一篇 20 頁論文，30 秒內看到導讀卡，每個導讀要點點擊後能跳到正確頁面並高亮。
2. 選取任一段文字提問，回答內容明確針對該段落。
3. 問一個文獻中沒有答案的問題，LLM 明確回答「文獻中未提及」而非編造。
4. 關閉瀏覽器重開，對話與高亮狀態完整保留。
